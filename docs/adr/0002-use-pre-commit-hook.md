# ADR 0002: ローカルフックを pre-push から pre-commit に切り替える

## ステータス

採用

## コンテキスト

このテンプレートでは、開発環境セットアップ時に `uv run pre-commit install` で Git hook を導入している。

これまでは、AI エージェントに任せるタスクを細かく分割し、人間が短いサイクルで頻繁にコミットする運用を想定していた。そのため、コミットのたびに待ち時間が発生しないよう、ローカルフックは `pre-push` に寄せていた。

一方で、GitHub Copilot CLI の導入により、以前より長めのまとまったタスクをエージェントへ委譲しやすくなった。結果として、コミット回数を抑えつつ、1コミットごとの意味と完成度を高めやすくなっている。

## 決定

`scripts/setup.sh` がインストールする Git hook を `pre-push` ではなく `pre-commit` に変更する。

具体的には、`uv run pre-commit install --hook-type pre-commit` を使って、コミット前に整形・静的検査・設定チェックが実行されるようにする。

## 理由

- コミット単位で問題を検出できるため、修正コミットの積み増しを減らしやすい。
- push 前にまとめて失敗するより、コミット前に止まる方が履歴を汚しにくい。
- Copilot CLI により長尺の実装タスクを任せやすくなり、従来ほど「人間が頻繁に小刻みにコミットする」前提に縛られなくなった。
- CI でも `uv run pre-commit run --all-files` を実行しており、ローカルの実行タイミングをコミット前へ寄せても整合性を取りやすい。

## 影響

- ローカルではコミット時の待ち時間が増える。
- その代わり、push 前やレビュー段階で発覚する軽微な不備は減ることが期待できる。
- セットアップ直後の開発体験としては、`scripts/setup.sh` 実行後にコミット前チェックが自動で有効になる。

## 代替案

### `pre-push` のまま維持する

コミット時の待ち時間は短いが、不備を含んだコミットがローカル履歴に残りやすい。

### `pre-commit` と `pre-push` を併用する

二重実行による待ち時間と運用コストが増えやすく、このテンプレートの初期設定としては過剰と判断した。
