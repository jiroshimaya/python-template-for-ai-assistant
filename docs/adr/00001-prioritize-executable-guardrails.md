# ADR 00001: Prioritize executable guardrails

- Status: accepted
- Date: 2026-03-21
- Supersedes: none
- Superseded by: none

## Context

このテンプレートは AI アシスタント、とくに Copilot CLI との協働を前提にしています。

2026-03-21 に、型チェック系の運用を `pyright` から `ty` に切り替える変更が入りました。実際には次のような段階的な調整が必要でした。

- `a7c5bb8`: `pyproject.toml`、`taskipy`、pre-commit hook を `pyright` から `ty` に置き換えた
- `e1bb0b9`: lockfile にまだ `pyright` が残っていたため、CI で一時的に `ty` を追加インストールする対応を入れた
- `ba0114d`: 利用可能な版に合わせて `ty==0.0.24` に固定し、CI の依存解決を安定させた
- `45c08f2`: 実行コマンドを `uv run ty check src` にそろえ、`pass_filenames: false` を加え、暫定 CI 対応を外した

この一連の変更は、「型チェッカーを変える」という一文だけでは十分に伝わりません。どのコマンドを標準とし、CI や pre-commit をどう一致させ、互換性問題にどう対処したのかまで含めて初めて、安全に再現できます。

AI エージェントに方針を伝える方法が README や instructions の説明文だけだと、背景と現在の有効性が混ざりやすく、古い判断が残ったままでも気付きにくくなります。

## Decision

このテンプレートでは、エージェント向けの重要な方針を決めるとき、説明文だけに依存せず、できる限り実行可能なガードレールを優先します。

具体的には次を基本方針とします。

- 方針の背景と採用理由は ADR として `docs/adr/` に記録する
- 実際の制約は、可能な限りテスト、CLI 手順、ディレクトリ規約、CI、lint 設定などの実行可能な形に落とし込む
- README や instructions は入口として使うが、長期的な判断根拠は ADR を正本とする
- ツール移行のように複数コミットへまたがる変更では、最終状態だけでなく、途中で露出した制約や暫定対応も ADR に要約して残す

## Consequences

判断理由を時系列で追いやすくなり、エージェントが古い説明文を根拠に誤作動するリスクを下げられます。

たとえば `pyright` から `ty` への移行では、「型チェッカーを変えた」という結論だけを見て `uv run ty` を実行すると、最終的に標準化された `uv run ty check src` や `pass_filenames: false` と食い違う危険がありました。ADR に経緯を残しておけば、エージェントは最終状態だけでなく、その状態に至る制約も踏まえて判断できます。

一方で、方針変更時には README や instructions の更新だけで済ませず、必要に応じて ADR の追加や superseded 更新も行う運用コストが増えます。

そのコストは、判断の履歴と有効な方針を明示できることによって回収できるとみなします。
